Poseable

2020 - STUDENT PROJECT - 4 WEEKS

Posable is a clear and simple tool for artists in need of anatomical reference while they work. It uses high quality, 3D models of humanoids in pre-set poses allowing the user to control camera angles and light settings to create the perfect scene to study.

Problem

I’ve used multiple apps that provide anatomical reference tools but I’ve always found them overwhelming. I wanted to put the focus on the pose itself while still giving the user the necessary tools to do their work.

Objective

Design one mobile application for both iOS and Android that provided identical feature sets and similar user flows.

Make each version feel native to their respective OS.

Role

UX Design
Wireframing
UI Design

Disclaimer: For the purpose of this exercise, all images of the 3D models used in the UI mockups were exported from the ArtPose iOS app.

Research

I started with a competitive analysis of 3 apps that provided the same basic feature set. Having used all of them myself at some point while illustrating, I knew where I wanted to go with my design goals.

My design needed to be minimilistic and stay out of the way.

Put the focus on the pose.


I focused on a small number of key features while creating the user flows. The other apps certainly had more bells and whistles, that may have been to their detriment.

Ideation and Wireframes

The main pain point I noticed from the competitors is that they have too many buttons available at all times. This created a cluttered UI that was hard to use on smaller screens. I aimed to fix this by only displaying options that are relevant to the mode the user is in.

Camera/View Mode:

The users gestures would rotate and pan the camera and contextual buttons would allow them to switch between a skin, muscle, and bone model view.

Light Mode:

The users gestures would rotate and pan the light source and contextual buttons would allow them to switch light colours and modes.

User Testing and Feedback

The testing was separated into 2 groups, one for each version of the application. To test how well the designs mimicked their respective home OS, I tried to make sure that iOS users tested the iOS version and vice versa.

I had a hard time visually communicating how the toggle and contextual options should work.

I had made it too minimal.

Iteration

I needed to rethink how I presented the camera/light toggle. Looking for inspiration, I landed on how iOS’ Phone app swaps between Missed Calls and All with a segmented control. Android doesn’t have a version of this that I could see so I attempted to design one that still felt at home on its OS.

I also toned down the contextual nature of the buttons, making them persistent and consolidating their toggles as best I could.

Summary

As stated a few times throughout this study, my initial attempts to make the design as minimal as possible really backfired on me. During my remote testing, nobody understood how the toggles were supposed to work, why buttons were disappearing and reappearing and where menus were. Once I was able to actually walk them through the process, it made sense to them but that shouldn’t be expected with a UI.

I also played it pretty safe and close to the design language of the respective operating systems here. Going forward, I would love to try and make something more unified that still felt at home on different devices and had a bit more personality.
